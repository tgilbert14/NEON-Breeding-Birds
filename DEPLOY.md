# Deploy & automation runbook

How the **NEON Breeding Bird Explorer** is hosted, kept current, and warmed up —
mirroring the Small Mammal Tracker / WaterChem flow.

```
  Posit Connect Cloud                          GitHub Pages  (docs/)
  └─ runs global/ui/server + R/ + www/         └─ landing page → "Launch" → app URL
     + data/*.rds indexes + climate tables        + og:image social card
     + data/sites/*.rds + data/env/*.rds          + fetch() pre-warm ping (cold-start fix)
     deploys straight from this GitHub repo       served from /docs on the default branch
     via manifest.json; re-publishes on push
```

Everything in this repo is **built and committed**. The deployed app runs entirely from the
bundled `.rds` files — `neonUtilities` is deliberately **not** a deploy dependency (it's referenced
by a computed name in `global.R` so the scanner never pins it; force bundle-only anywhere with
`BRD_LIVE=0`). The landing page already exists at [`docs/index.html`](docs/index.html) and the
social card at [`docs/og-image.png`](docs/og-image.png).

---

## Option A — Posit Connect Cloud (recommended; works today)

Connect Cloud deploys an R app straight from a GitHub repo using `manifest.json` (generated by
[`scripts/write_manifest.R`](scripts/write_manifest.R) — lean, bundle-only, 91 CRAN packages, **no
`neonUtilities`**).

1. Sign in at <https://connect.posit.cloud> with GitHub.
2. **New Content → Shiny → from Git**, pick `tgilbert14/NEON-Breeding-Birds`, branch `master`.
   Connect reads `manifest.json` (`appmode: shiny`) and auto-detects the multi-file app
   (`global.R` + `ui.R` + `server.R`) — there's no single "primary file" to choose; `global.R`
   is sourced automatically.
3. It restores packages and serves. **Auto-republishes on every push to `master`.**
4. Copy the published share URL → set `APP_URL` near the bottom of [`docs/index.html`](docs/index.html)
   (and the README badge) → re-publish Pages.

Regenerate the manifest whenever runtime dependencies change, then commit it:
```r
# R with the app's runtime packages (R 4.3.1 has them all):
Rscript scripts/write_manifest.R     # writes manifest.json + self-checks neonUtilities is absent
```

**Cold start:** the free tier sleeps. The landing page's pre-warm `fetch(APP_URL)` on load wakes
it while the visitor reads, so the app is usually warm by the time they click **Launch**.

## Option B — Shinylive / WebAssembly (possible later: static, zero server)

The deployed app has no hard runtime dependency on `neonUtilities`, so it's a shinylive candidate.
**Gates:** (1) every package needs a wasm binary — verify each (esp. **`leaflet`**) at
<https://repo.r-wasm.org/>; (2) the payload is heavier than the mammal app — 46 site bundles **plus**
46 env overlays + the climate tables ship to the browser, so confirm the size is acceptable before
committing. If clean: `shinylive::export(".", "docs/app")` and set `APP_URL = "app/"` (one Pages
deploy: landing + app). If anything fails, stay on Connect Cloud.

## Publish the landing page (GitHub Pages)

1. **The repo must be public** for free GitHub Pages (or have GitHub Pro/Team for private Pages).
2. Repo **Settings → Pages → Source: Deploy from a branch → `master` / `/docs`**.
3. Live at `https://tgilbert14.github.io/NEON-Breeding-Birds/`. The `og:image` meta already uses
   this URL. `docs/.nojekyll` is present so Pages serves the files as-is.
4. Regenerate the card after a copy/stat change: `Rscript scripts/make_og_image.R`.

## Keeping the data current (automatic)

[`.github/workflows/refresh-data.yml`](.github/workflows/refresh-data.yml) — tuned to each
product's cadence (birds publish ~annually; sensors monthly):

- **Monthly** (first Saturday night, Arizona): a light **env top-up** (precip/temp/phenology, last
  ~14 months) + a rebuild of the climate precompute tables → push to `master` → Connect Cloud
  re-publishes. Fast and low-risk (never drops sites).
- **On demand**: tick **`refetch_birds`** in the Actions tab to re-pull all 46 bird sites when NEON
  publishes a new breeding season (guarded — only re-bundles when ≥40 sites are fetched).
- Optional repo secret **`NEON_TOKEN`** (Settings → Secrets and variables → Actions) raises the API
  rate limit; the pulls work without it, just slower.

Rebuild the whole bundle from scratch locally (R-4.1.1 for the NEON pulls; R-4.5.x crashes on
`loadByProduct`):
```
1. Rscript scripts/fetch_bird_all.R          # all 46 sites -> ../bird-data-fetch
2. Rscript scripts/bundle_bird_data.R        # -> data/sites/*.rds + data/site_index.rds
3. Rscript scripts/refresh_env_data.R        # -> data/env/*.rds  (full 13-yr build, once)
4. Rscript scripts/refresh_site_climate.R    # -> data/site_climate.rds + data/site_month_clim.rds
5. Rscript scripts/build_cross_site.R        # -> data/cross_site.rds (effort-rarefied metrics)
6. Rscript scripts/write_manifest.R          # -> manifest.json (commit it)
```

## What you (the human) need to do — first deploy

1. **Make the repo public** (Settings → General → Change visibility) if you want the free Pages
   landing page. Connect Cloud itself can use a private repo, but Pages free cannot.
2. **Deploy on Connect Cloud** (Option A, steps 1-3) and copy the share URL.
3. **Set `APP_URL`** in `docs/index.html` (and the README badge) to that URL; commit.
4. **Enable Pages** (Settings → Pages → `master` / `/docs`).
5. *(optional)* Add the **`NEON_TOKEN`** Actions secret for faster scheduled refreshes.

After step 3, the landing page's Launch button works and pre-warms the app on every visit.
